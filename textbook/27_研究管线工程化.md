## 详细讲解

### 1. 为什么量化研究一定要工程化

前面的课程已经做了很多策略：

```text
均线趋势；
突破；
均值回归；
因子；
配对交易；
机器学习信号。
```

一开始用 notebook 很方便，因为你要快速看图、试参数、理解概念。

但策略越来越多之后，纯 notebook 会出现几个问题：

```text
不知道某张图是用哪组参数生成的；
不知道数据有没有更新；
不知道成本假设有没有改；
不知道结果能不能重新跑出来；
复制粘贴代码时容易漏改变量；
过几周回来已经忘了当时怎么做的。
```

所以第 27 章开始进入工程化阶段。

工程化不是为了把代码写得复杂，而是为了让研究结果可复现。

### 2. 研究管线是什么

本章关键代码写成：

```python
config -> data -> signal -> backtest -> report -> artifacts
```

这其实就是一个标准研究流程。

可以拆成五步：

```text
config：记录这次实验的参数；
data：加载并固定数据；
signal：根据规则生成信号；
backtest：把信号转成收益和指标；
report：输出表格、图和结论；
artifacts：保存所有可复查产物。
```

你可以把它理解成量化研究的生产线。

每次输入固定的配置，管线就应该生成同一套结果。

### 3. config 为什么重要

配置记录的是实验条件。

本章配置包括：

```text
symbols = SPY, QQQ, DIA, IWM, EFA, TLT
start = 2006-01-03
end = 2026-05-18
strategy = MA 10/200 band 1%
cost_bps = 3.0
```

这些信息看起来普通，但非常关键。

没有配置，你以后看到结果时会不知道：

```text
用了哪些资产；
从哪天开始；
到哪天结束；
均线参数是多少；
有没有扣交易成本；
成本是多少 bps。
```

量化研究里，结论永远依赖条件。

所以每个结论都必须能追溯到对应配置。

### 4. data 为什么要固定

数据是策略研究的地基。

如果今天下载一次数据，明天又下载一次数据，中间可能出现：

```text
复权调整变化；
缺失值修复；
数据源字段变化；
最新交易日增加；
历史数据被供应商修正。
```

这些变化可能让回测结果不同。

所以工程化管线里通常要有数据缓存和数据版本意识。

本章不是完整数据平台，但已经开始强调：

```text
同一份研究结果，要知道它用的是哪一段数据。
```

这也是为什么报告里记录：

```text
rows = 5125
start_date = 2006-01-03
end_date = 2026-05-18
```

### 5. signal 层负责什么

signal 层只负责回答一个问题：

```text
今天每个资产应该持有还是不持有？
```

本章复用的是多资产趋势策略：

```text
MA 10/200 band 1%
```

它会对 SPY、QQQ、DIA、IWM、EFA、TLT 分别产生趋势信号。

signal 层最好保持单纯：

```text
输入价格；
输出信号。
```

不要在 signal 层混入报告输出、画图、路径保存等逻辑。

这样以后你替换策略时，只需要换 signal 模块，其他流程还可以复用。

### 6. backtest 层负责什么

backtest 层负责把信号变成组合收益。

它要处理：

```text
信号滞后；
开盘成交还是收盘成交；
持仓权重；
换手率；
交易成本；
组合净值；
回撤和风险指标。
```

这层是研究管线的核心，因为信号本身不等于收益。

一个信号看起来很准，但如果：

```text
换手太高；
成本太高；
成交假设不真实；
仓位计算有误；
```

最终也可能不能交易。

### 7. report 和 artifacts 有什么区别

report 是给人看的。

它应该回答：

```text
这次实验做了什么；
用了什么数据和参数；
结果怎么样；
有什么风险；
结论是什么。
```

artifacts 是给复查和复现用的产物。

它可以包括：

```text
图像；
表格；
参数文件；
日志；
中间结果；
最终报告。
```

以后你要比较两个策略，不应该靠记忆，而应该靠这些产物。

### 8. 本章实跑检查怎么读

本章结果表不是普通绩效表，而是 pipeline check：

```text
data_rows = 5125，pass
symbols = SPY,QQQ,DIA,IWM,EFA,TLT，pass
parameters_recorded = MA 10/200 band 1%，pass
cost_recorded = 3.0，pass
reference_final_equity = 4.7201，pass
```

这里的重点不是说 final_equity 4.7201 一定很好。

重点是：

```text
数据行数被记录；
资产列表被记录；
参数被记录；
成本被记录；
参考结果被记录。
```

以后如果你改了代码，reference_final_equity 从 4.7201 变成 4.10 或 5.30，你就要问：

```text
我是有意改了策略逻辑？
还是不小心改坏了回测？
```

这就是工程化给你的保护。

### 9. reference_final_equity 不是优化目标

这里要特别注意。

reference_final_equity 不是让你不断把 4.7201 优化得更高。

它更像回归测试里的参考值：

```text
如果代码逻辑没变，结果应该大致一致。
```

如果你改变了策略规则、成本、成交假设或数据区间，结果当然可以变化。

但变化必须可解释。

工程化的目标不是让回测更漂亮，而是让变化可追踪。

### 10. 个人量化为什么更需要工程化

个人开发者最容易犯的错误不是不会写复杂模型，而是：

```text
研究没有记录；
参数随手改；
结果无法复现；
亏钱后找不到原因；
策略越来越多但没有统一标准。
```

机构有研究平台、代码审查、数据团队和风控流程。

个人没有这些外部约束，所以更要用工程化约束自己。

对你来说，工程化的最低标准是：

```text
每个策略有脚本；
每次实跑有报告；
关键参数写进配置；
图和结果保存到固定目录；
测试能检查核心逻辑没有被意外改坏。
```

### 11. 本章过关标准

你能讲清楚下面四句话，第 27 章就算过关：

```text
研究管线把一次策略实验拆成配置、数据、信号、回测、报告和产物。
工程化的核心价值是可复现、可审计、可比较。
reference result 用来发现代码变化，不是用来盲目优化收益。
个人量化更需要工程纪律，因为没有机构流程替你兜底。
```
